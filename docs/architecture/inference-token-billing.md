inference token consumption for LLM workloads running on
OpenShift AI (RHOAI), enabling operators and enterprises to
perform chargeback and showback for AI inference usage.

## Overview

### What is This Feature?

Inference token billing extends Koku's OpenShift cost
management to track per-tenant, per-model LLM inference
consumption measured in tokens (input and output). This
complements the existing GPU billing feature, which tracks
GPU uptime and MIG slice allocation.

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Input tokens** | Tokens in the prompt sent to the model (also called prompt tokens) |
| **Output tokens** | Tokens generated by the model (also called completion/generation tokens) |
| **Model name** | The LLM being served (e.g., `llama-3-70b`, `mistral-7b`) |
| **Token rate** | Price per token (e.g., $0.01 per 1K input tokens, $0.03 per 1K output tokens) |
| **vLLM** | Default inference engine in Red Hat OpenShift AI |

### Scope

- **In scope:** vLLM token counters on OpenShift (via
  Prometheus), cost model rates, report API, on-prem support
- **Out of scope:** Real-time token quota enforcement
  (handled by Red Hat Connectivity Link's
  `TokenRateLimitPolicy`), billing/invoicing (operator
  responsibility)

### Use Cases

| User | Need | How Koku Helps |
|------|------|----------------|
| **GPU cloud operator** | Bill tenants for inference usage | Token counts + cost model → `/reports/openshift/inference-tokens/` |
| **Enterprise FinOps** | Chargeback across business units | Group by namespace (team) × model → cost attribution |
| **Platform engineer** | Capacity planning | Token throughput trends → when to add GPUs |

## Why Token Billing

Traditional GPU billing charges for **time** (GPU-hours).
This works for training but not for inference:

- A tenant running a 7B model uses far fewer GPU resources
  per hour than a tenant running a 70B model
- A tenant sending 1K requests/hour should pay more than a
  tenant sending 10 requests/hour on the same GPU
- Request-per-second rate limiting does not reflect actual
  compute cost — a single prompt can consume thousands of
  tokens

Token billing charges for **work done**, not time occupied.
It complements GPU-hour billing:

| Billing metric | What it measures | Best for |
|---------------|-----------------|----------|
| GPU-hours | Time a GPU was allocated | Training, fine-tuning, reservation-based billing |
| Inference tokens | Work performed by the model | Inference, pay-per-use billing |

## Data Flow

```
vLLM pods (RHOAI)
  │  Prometheus counters:
  │  vllm:prompt_tokens_total
  │  vllm:generation_tokens_total
  ▼
koku-metrics-operator
  │  Queries Prometheus hourly
  │  Produces CSV: cm-openshift-inference-token-usage-YYYYMM.csv
  ▼
Koku backend
  │  Parquet ingestion → daily line items
  │  Trino/PostgreSQL aggregation → summary table
  │  Cost model rates applied
  ▼
API: GET /reports/openshift/inference-tokens/
  │  Group by: namespace, model_name, inference_service, organization, cluster
  │  Filter by: model_name, inference_service, organization, namespace, date range
  ▼
Console / FinOps dashboard / billing system
```

## Metrics Collection

### Prometheus Queries (koku-metrics-operator)

The operator supports two metric sources with automatic
fallback: vLLM-native metrics (primary) and OTel GenAI
standard metrics (fallback).

**vLLM-native (primary):**

```promql
# Input tokens — vLLM with KServe InferenceService label
sum by (exported_pod, exported_namespace, Hostname, model_name,
        label_serving_kserve_io_inferenceservice)(
  increase(vllm:prompt_tokens_total[1h])
  * on(exported_pod, exported_namespace)
    group_left(label_serving_kserve_io_inferenceservice)
    label_replace(label_replace(kube_pod_labels, ...))
)
```

**OTel GenAI standard (fallback):**

```promql
# Input tokens — OpenTelemetry gen_ai.client.token.usage
sum by (namespace, pod, gen_ai_request_model)(
  gen_ai_client_token_usage_sum{gen_ai_token_type="input"}
)
```

The operator tries vLLM metrics first. If no results are
returned, it falls back to OTel GenAI metrics. Sources
for OTel GenAI metrics include MLflow (3.2+ with
`MLFLOW_ENABLE_OTEL_GENAI_SEMCONV=true`), Envoy AI
Gateway, and any OTel-instrumented inference stack.

Both sources produce the same CSV format — the Koku
backend processes them identically.

### CSV Report Format

The operator produces a CSV file per month:

| Column | Type | Description |
|--------|------|-------------|
| `report_period_start` | timestamp | Reporting period start |
| `report_period_end` | timestamp | Reporting period end |
| `interval_start` | timestamp | Collection interval start |
| `interval_end` | timestamp | Collection interval end |
| `node` | string | Node hostname |
| `namespace` | string | Kubernetes namespace |
| `pod` | string | Pod name |
| `model_name` | string | LLM model identifier |
| `inference_service` | string | KServe InferenceService name (from `serving.kserve.io/inferenceservice` pod label) |
| `organization` | string | Organization identity (from X-Organization header or metric label) |
| `input_tokens` | decimal | Prompt tokens consumed |
| `output_tokens` | decimal | Generation tokens produced |

File naming: `cm-openshift-inference-token-usage-YYYYMM.csv`

**Model:** `OCPInferenceTokenSummaryP`

**Table:** `reporting_ocp_inference_token_summary_p`
(partitioned by `usage_start`)

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |

| `cluster_alias` | varchar | Cluster display name |
| `usage_start` | date | Day of usage |
| `usage_end` | date | End of usage period |
| `namespace` | varchar | Kubernetes namespace (tenant) |
| `node` | varchar | Node hostname |
| `model_name` | varchar | LLM model name |
| `inference_service` | varchar | KServe InferenceService name |
| `organization` | varchar | Organization identity (Keycloak realm / OSAC org) |
| `input_tokens` | decimal | Total input tokens for the day |
| `output_tokens` | decimal | Total output tokens for the day |
| `total_tokens` | decimal | input + output tokens |
| `cost_model_inference_cost` | decimal | Calculated cost |
| `cost_model_rate_type` | text | Infrastructure or Supplementary |
| `source_uuid` | UUID | Source provider reference |

**Indexes:** `usage_start`, `cluster_id`, `namespace`,
`model_name`

Data is stored at **daily granularity**. The API supports
querying any time range — daily, weekly, monthly, or custom
date ranges.

## Cost Calculation

### Cost Model Metrics

Two new metrics are added to the OCP cost model:

| Metric constant | Rate key | Unit | Description |
|-----------------|----------|------|-------------|
| `OCP_INFERENCE_INPUT_TOKEN` | `inference_input_token_cost_per_token` | tokens | Rate per input token |
| `OCP_INFERENCE_OUTPUT_TOKEN` | `inference_output_token_cost_per_token` | tokens | Rate per output token |

Unlike GPU rates (monthly, amortized daily) or CPU rates
(per core-hour), token rates are **per-unit**: the cost
equals `tokens × rate` with no time-based amortization.
This reflects the industry-standard model where output
tokens are typically priced higher than input tokens.

### Cost Formula

```
daily_cost = (input_tokens × input_rate) + (output_tokens × output_rate)
```

### Example

A namespace consumed 1M input tokens and 500K output tokens
in a day. The cost model defines:
- Input rate: $0.00001 per token ($10 per 1M tokens)
- Output rate: $0.00003 per token ($30 per 1M tokens)

```
daily_cost = (1,000,000 × $0.00001) + (500,000 × $0.00003)
           = $10.00 + $15.00
           = $25.00
```

The API returns this at whatever time range is requested.
A 30-day report sums daily costs:

```
monthly_cost = Σ daily_cost = $25.00 × 30 = $750.00
```

## Identity-Based Billing (Sovereign Gateway Pattern)

For shared model endpoints where multiple organizations
share the same inference service, tokens are attributed by
**organization identity** rather than by namespace. This
uses the Sovereign Gateway Pattern:

```
User (Keycloak JWT)
  │
  ▼
Authorino (token validation)
  │  Extracts: organization from JWT claims
  │  Sets: X-Organization header
  ▼
Envoy AI Gateway / vLLM
  │  Serves inference request
  ▼
TelemetryPolicy (Istio)
  │  Emits: authorized_hits{organization="acme"} metric
  ▼
koku-metrics-operator
  │  Collects per-organization token counts
  │  CSV columns: organization
  ▼
Koku
  │  group_by[organization] → per-org cost reports
  ▼
Billing / Chargeback
```

### How It Works

1. **Keycloak JWT** — Users authenticate via Keycloak
   (or any OIDC provider). The JWT contains `organization`
   (realm or custom claim).

2. **Authorino** — Validates the JWT and extracts the
   organization claim into request headers
   (`X-Organization`). This is a Red Hat Connectivity Link
   component.

3. **TelemetryPolicy** — An Istio telemetry policy emits
   `authorized_hits` metrics with `organization` labels,
   counting tokens per organization.

4. **koku-metrics-operator** — Queries Prometheus for
   per-organization token counts and includes `organization`
   in the CSV report.

5. **Koku** — Processes the CSV, populates the summary
   table with `organization` field, and exposes it as a
   group-by/filter dimension in the API.

### Relationship to OSAC

The `organization` field in token billing data maps
directly to OSAC Organizations / Keycloak realms. When
identity-based metrics are used, the organization IS the
tenant identity — no additional tenant resolution layer
is needed. See
[Tenant Provider Model](tenant-provider-model.md)
for details.

## SLA-Based Credits (Tiered Billing)

### Overview

When inference latency degrades, token costs are
automatically discounted using three TTFT (time-to-first-token)
latency buckets. Each bucket carries a configurable billing
fraction, giving operators fine-grained control over SLA
credits. This creates a direct financial incentive for
platform operators to maintain inference quality.

### Three-Bucket Model

The koku-metrics-operator classifies each request into
one of three latency tiers using the vLLM
`vllm:time_to_first_token_seconds_bucket` histogram:

| Bucket | TTFT Range | Billing Fraction | Meaning |
|--------|-----------|-------------------|---------|
| `sla_good` | < 500 ms | 1.0 (full price) | Requests met SLA |
| `sla_degraded` | 500 ms - 2 s | 0.5 (50% discount) | Requests were slow |
| `sla_breached` | >= 2 s | 0.0 (free) | Requests violated SLA |

The three fractions always sum to 1.0 for a given
collection interval. The `sla_compliance` field is
retained for backward compatibility (it equals
`sla_good`).

### How Discounts Are Applied

SLA discounts use Koku's existing **negative markup**
mechanism. The three-bucket model gives operators
visibility into latency distribution. The operator
reviews the bucket data and configures markup:

| Scenario | sla_good | sla_degraded | sla_breached | Operator action |
|----------|----------|--------------|--------------|-----------------|
| Excellent | 0.95 | 0.04 | 0.01 | No adjustment |
| Degraded | 0.70 | 0.20 | 0.10 | Set markup = -20% |
| Severe | 0.40 | 0.30 | 0.30 | Set markup = -50% |

This approach:
- Reuses the existing markup mechanism (no new code)
- Gives operators control over discount amounts
- Provides per-bucket visibility for informed decisions
- Avoids hardcoded discount factors in SQL

### Implementation Notes

- **Reporting only:** Bucket fields are visible in the
  API for operators to make discount decisions. They do
  not automatically adjust cost.
- **Aggregation:** All three bucket fields are averaged
  (not summed) when rolling up across time periods.
- **API:** All three fields are exposed as `Avg` annotations
  in the report API alongside `sla_compliance`.

## On-Premise Support

Koku supports on-premise deployment (`ONPREM=True`). For
inference tokens:

| Component | SaaS | On-prem |
|-----------|------|---------|
| Data processing | Trino (Hive/S3) | PostgreSQL directly |
| Summary SQL | `trino_sql/...` | `self_hosted_sql/...` |
| Feature flags | Unleash server | `MockUnleashClient` (enabled by default) |
| Data source | Operator uploads to console.redhat.com | Operator stores locally (air-gapped) |

Both SQL variants produce identical summary data in the
`reporting_ocp_inference_token_summary_p` table.

## Relationship to GPU Billing

Inference token billing is **complementary** to GPU billing,
not a replacement:

| | GPU billing | Token billing |
|---|---|---|
| **Metric** | GPU uptime (seconds) | Tokens consumed (count) |
| **Rate type** | Monthly (amortized daily) | Per-token (direct multiplication) |
| **Best for** | Training, fine-tuning, reserved capacity | Inference, pay-per-use |
| **Summary table** | `reporting_ocp_gpu_summary_p` | `reporting_ocp_inference_token_summary_p` |
| **Cost model metric** | `gpu_cost_per_month` | `inference_input/output_token_cost_per_token` |
| **MIG awareness** | Yes (slice-weighted cost) | No (tokens are model-level) |

A complete billing model uses **both**: GPU-hours for
capacity reservation and tokens for usage-based pricing.
